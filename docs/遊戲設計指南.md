# 遊戲設計指南（HTML5 單檔小遊戲）

> 本指南是本 repo（AI未來創作營隊）所有遊戲的設計聖經。每次開發新遊戲前先讀完本文。
> 目標品質標準：**瑪利歐等級的手感與設計**。技術限制：單一 HTML 檔、vanilla JS + canvas、零外部資源。

---

## (a) 設計原則檢查表 —「瑪利歐標準」

每個遊戲出貨前，以下四項缺一不可：

| 原則 | 意義 | 自我檢驗問題 |
|------|------|--------------|
| **即時回饋** | 每個玩家輸入在 <100ms 內有視覺+聽覺回應 | 按下按鍵時，畫面上有東西動了嗎？有聲音嗎？ |
| **漸進難度** | 先教學、再變化、再考驗（見第 (d)(4) 節起承轉合） | 前 30 秒不用說明文字，玩家能學會操作嗎？ |
| **失敗不挫折** | 死亡後 <1 秒重來、關卡短、有檢查點、失敗原因一目了然 | 死掉時玩家會說「再一次！」還是「好煩」？ |
| **一眼看懂** | 主角/敵人/道具/背景四層視覺清楚分離，剪影可辨識 | 截圖縮到 1/4 大小，還分得出誰是誰嗎？ |

補充原則：

- **寬容設計（Forgiveness）**：所有判定往玩家有利的方向作弊——碰撞箱比圖形小、時間窗比直覺寬。參考 [Celeste & Forgiveness](https://maddythorson.medium.com/celeste-forgiveness-31e4a40399f1)。
- **一個核心機制**：一款遊戲只圍繞一個動詞（跳、切、閃、疊…），把它做到極致，而非堆功能。
- **難度曲線鋸齒狀**：每次高峰後給一段輕鬆段落當「喘息」，再拉更高。
- **獎勵大聲、懲罰小聲**：吃金幣要煙火，掉坑只要短促音效＋快速重生。

## (b) 程式架構模板

所有新遊戲以此骨架起手（固定時間步進 + 狀態機 + 鍵盤/觸控輸入 + 自適應縮放）。
固定時間步進做法源自 [Fix Your Timestep! — Gaffer On Games](https://gafferongames.com/post/fix-your-timestep/)：物理以固定 dt 更新確保跨裝置行為一致，渲染則每幀執行。

```html
<!doctype html>
<html lang="zh-Hant">
<head>
<meta charset="utf-8">
<meta name="viewport" content="width=device-width,initial-scale=1,user-scalable=no,viewport-fit=cover">
<title>遊戲名稱</title>
<style>
  html,body{margin:0;height:100%;background:#1a1c2c;overflow:hidden;touch-action:none;
    -webkit-user-select:none;user-select:none}
  canvas{position:absolute;left:50%;top:50%;transform:translate(-50%,-50%)}
</style>
</head>
<body>
<canvas id="game"></canvas>
<script>
'use strict';
// ===== 基本設定：遊戲以固定邏輯解析度設計，再等比縮放到螢幕 =====
const GAME_W = 960, GAME_H = 540;          // 邏輯解析度（16:9）
const canvas = document.getElementById('game');
const ctx = canvas.getContext('2d');

// ===== 自適應縮放（含 devicePixelRatio，確保高解析螢幕不模糊）=====
function resize(){
  const scale = Math.min(innerWidth / GAME_W, innerHeight / GAME_H);
  const dpr = Math.min(devicePixelRatio || 1, 2);      // 上限 2，避免低階機過熱
  canvas.width  = GAME_W * scale * dpr;
  canvas.height = GAME_H * scale * dpr;
  canvas.style.width  = GAME_W * scale + 'px';
  canvas.style.height = GAME_H * scale + 'px';
  ctx.setTransform(scale * dpr, 0, 0, scale * dpr, 0, 0); // 之後全用邏輯座標繪圖
  canvas._scale = scale;                                  // 觸控座標換算用
}
addEventListener('resize', resize); resize();

// ===== 輸入：鍵盤 + 觸控統一成抽象動作 =====
const input = { left:false, right:false, action:false, actionBuffer:0 };
const KEYMAP = { ArrowLeft:'left', KeyA:'left', ArrowRight:'right', KeyD:'right',
                 Space:'action', ArrowUp:'action', KeyW:'action' };
addEventListener('keydown', e => {
  const a = KEYMAP[e.code]; if(!a) return; e.preventDefault();
  if(a === 'action' && !input.action) input.actionBuffer = 0.12; // 跳躍緩衝 120ms
  input[a] = true;
});
addEventListener('keyup', e => { const a = KEYMAP[e.code]; if(a) input[a] = false; });

// 觸控：左半螢幕左右移動、右半螢幕動作鍵（依遊戲類型調整，見第 (f) 節）
function touchPos(t){
  const r = canvas.getBoundingClientRect();
  return { x:(t.clientX - r.left) / canvas._scale, y:(t.clientY - r.top) / canvas._scale };
}
function handleTouches(e){
  e.preventDefault();
  const wasAction = input.action;
  input.left = input.right = input.action = false;
  for(const t of e.touches){
    const p = touchPos(t);
    if(p.x < GAME_W * 0.25) input.left = true;
    else if(p.x < GAME_W * 0.5) input.right = true;
    else input.action = true;
  }
  if(input.action && !wasAction) input.actionBuffer = 0.12;
}
for(const ev of ['touchstart','touchmove','touchend','touchcancel'])
  canvas.addEventListener(ev, handleTouches, { passive:false });

// ===== 音效：Web Audio 合成（首次觸碰才建立，符合瀏覽器自動播放政策）=====
let AC = null;
function audio(){ if(!AC) AC = new (window.AudioContext||window.webkitAudioContext)(); return AC; }
addEventListener('pointerdown', audio, { once:true });
addEventListener('keydown', audio, { once:true });

// ===== 狀態機：每個狀態自帶 update / draw =====
let state;                      // 目前狀態物件
function setState(s){ state = s; if(s.enter) s.enter(); }

const menuState = {
  update(dt){ if(input.actionBuffer > 0){ input.actionBuffer = 0; setState(playState); } },
  draw(){
    ctx.fillStyle = '#1a1c2c'; ctx.fillRect(0,0,GAME_W,GAME_H);
    ctx.fillStyle = '#f4f4f4'; ctx.textAlign = 'center';
    ctx.font = 'bold 48px sans-serif'; ctx.fillText('遊戲名稱', GAME_W/2, 220);
    ctx.font = '24px sans-serif';
    ctx.fillText('按空白鍵 / 點擊螢幕開始', GAME_W/2, 300);
  }
};

const playState = {
  enter(){ /* 重設玩家、關卡、分數 */ },
  update(dt){
    // 遊戲邏輯放這裡（玩家物理、敵人、碰撞、計分）
  },
  draw(){
    ctx.fillStyle = '#1a1c2c'; ctx.fillRect(0,0,GAME_W,GAME_H);
    // 繪製順序：遠景 → 近景 → 實體 → 玩家 → 粒子 → UI
  }
};

const gameOverState = {
  enter(){ this.t = 0; },
  update(dt){
    this.t += dt;                                        // 強制冷卻 0.5 秒防誤觸
    if(this.t > 0.5 && input.actionBuffer > 0){ input.actionBuffer = 0; setState(playState); }
  },
  draw(){
    playState.draw();                                    // 背景維持死亡瞬間畫面
    ctx.fillStyle = 'rgba(0,0,0,.55)'; ctx.fillRect(0,0,GAME_W,GAME_H);
    ctx.fillStyle = '#f4f4f4'; ctx.textAlign = 'center';
    ctx.font = 'bold 40px sans-serif'; ctx.fillText('再挑戰一次！', GAME_W/2, 260);
  }
};

// ===== 主迴圈：固定時間步進（60Hz 物理）＋每幀渲染 =====
const STEP = 1/60;
let last = performance.now(), acc = 0;
function frame(now){
  requestAnimationFrame(frame);
  acc += Math.min((now - last) / 1000, 0.1);  // 上限 0.1s：切分頁回來不會瞬移
  last = now;
  while(acc >= STEP){
    input.actionBuffer = Math.max(0, input.actionBuffer - STEP);
    state.update(STEP);
    acc -= STEP;
  }
  state.draw();
}
setState(menuState);
requestAnimationFrame(frame);
</script>
</body>
</html>
```

實體管理保持簡單即可：一個 `entities` 陣列，每個實體是有 `update(dt)`/`draw()` 的普通物件，死亡標記 `dead:true` 後用 `entities = entities.filter(e => !e.dead)` 清除。**不需要** ECS 框架。

## (c) 手感（Game Feel / Juice）配方

核心觀念來自 [Juice it or lose it（Martin Jonasson & Petri Purho）](https://www.youtube.com/watch?v=Fy0aCDmgnxg)與 [The Art of Screenshake（Jan Willem Nijman / Vlambeer）](https://www.youtube.com/watch?v=AJdEqssNZ-U)：同一份遊戲邏輯，加上回饋層之後好玩十倍。每個重要事件（跳躍、命中、得分、死亡）都應該同時觸發 2~3 種下列效果。

### 1. 螢幕震動（Screen Shake）

```js
let shake = 0;                                  // 呼叫 shake = 8 觸發（像素強度）
// update 內：
shake *= Math.pow(0.001, dt);                   // 指數衰減，約 0.3 秒歸零
// draw 開頭：
ctx.save();
ctx.translate((Math.random()*2-1)*shake, (Math.random()*2-1)*shake);
// ...繪製世界...
ctx.restore();                                  // UI 在 restore 之後畫，不跟著震
```
用量：小命中 3px、爆炸 8px、Boss 登場 14px。**寧少勿多**，持續震動會頭暈。

### 2. 命中停頓（Hit-stop / Freeze Frame）

```js
let freeze = 0;                                 // 呼叫 freeze = 0.06 觸發（秒）
// 主迴圈 while 內最前面：
if(freeze > 0){ freeze -= STEP; continue; }     // 跳過邏輯更新但照常渲染
```
用量：普通命中 0.03~0.06s、關鍵一擊 0.1~0.15s。讓打擊「有重量」的最便宜手法。

### 3. 壓扁與伸展（Squash & Stretch）

```js
// 玩家物件加 sx, sy（縮放），每幀彈回 1：
p.sx += (1 - p.sx) * 12 * dt;  p.sy += (1 - p.sy) * 12 * dt;
// 起跳瞬間：p.sx = 0.7; p.sy = 1.4;   落地瞬間：p.sx = 1.4; p.sy = 0.6;
// 繪製時以腳底為錨點縮放：
ctx.save();
ctx.translate(p.x, p.y + p.h);                  // 錨點在腳底，壓扁時不會浮空
ctx.scale(p.sx, p.sy);
drawPlayer(-p.w/2, -p.h);                       // 用區域座標畫角色
ctx.restore();
```

### 4. 粒子系統（30 行夠用）

```js
const parts = [];
function burst(x, y, color, n = 12, speed = 220){
  for(let i = 0; i < n; i++){
    const a = Math.random() * Math.PI * 2, s = speed * (0.3 + Math.random() * 0.7);
    parts.push({ x, y, vx:Math.cos(a)*s, vy:Math.sin(a)*s, life:0.5, color });
  }
}
function updateParts(dt){
  for(const p of parts){
    p.x += p.vx * dt; p.y += p.vy * dt; p.vy += 600 * dt; p.life -= dt;
  }
  for(let i = parts.length - 1; i >= 0; i--) if(parts[i].life <= 0) parts.splice(i, 1);
}
function drawParts(){
  for(const p of parts){
    ctx.globalAlpha = Math.max(0, p.life * 2);
    ctx.fillStyle = p.color;
    ctx.fillRect(p.x - 3, p.y - 3, 6, 6);       // 方形粒子最有像素風味
  }
  ctx.globalAlpha = 1;
}
```
必加場合：落地（腳下小塵土 4 顆）、吃道具（金色 12 顆）、死亡（大爆 24 顆）。

### 5. 緩動（Easing / Tween）

UI 與鏡頭**永遠不要瞬移**。兩個萬用公式（參考 [Easing Functions Cheat Sheet](https://easings.net/)）：

```js
// 追隨式（每幀朝目標靠近，適合鏡頭、血條、分數滾動）
cam.x += (target.x - cam.x) * (1 - Math.pow(0.01, dt));   // 幀率無關的平滑追隨
// 時間式（0→1 的 t，適合選單進出場）
const easeOutBack  = t => 1 + 2.7 * Math.pow(t-1, 3) + 1.7 * Math.pow(t-1, 2); // 過衝回彈
const easeOutCubic = t => 1 - Math.pow(1 - t, 3);
```

### 6. 音效合成（Web Audio API，零音檔）

```js
function beep(freq = 440, dur = 0.1, type = 'square', vol = 0.15, slide = 0){
  const ac = audio(); if(!ac) return;
  const o = ac.createOscillator(), g = ac.createGain();
  o.type = type; o.frequency.setValueAtTime(freq, ac.currentTime);
  if(slide) o.frequency.exponentialRampToValueAtTime(
    Math.max(30, freq + slide), ac.currentTime + dur);
  g.gain.setValueAtTime(vol, ac.currentTime);
  g.gain.exponentialRampToValueAtTime(0.001, ac.currentTime + dur); // 收尾避免爆音
  o.connect(g).connect(ac.destination);
  o.start(); o.stop(ac.currentTime + dur);
}
// 常用配方：
// 跳躍:   beep(200, 0.15, 'square', 0.12, +400)   上滑音
// 吃金幣: beep(988, 0.08, 'square'); setTimeout(() => beep(1319, 0.15, 'square'), 80)
// 受傷:   beep(220, 0.25, 'sawtooth', 0.2, -150)  下滑音
// 落地:   beep(120, 0.06, 'triangle', 0.1)
// 爆炸（噪音）：用 AudioBuffer 填入 Math.random()*2-1，接 gain 衰減播放
```

## (d) 平台跳躍物理參數表

### 1. 由「設計值」反推物理參數

不要憑感覺調重力。先決定跳多高、多久到頂點，再套公式（[GDQuest 跳躍公式](https://www.gdquest.com/library/kinematic_jump_formulas/)、[Platformer Physics 101](https://error454.com/2013/10/23/platformer-physics-101-and-the-3-fundamental-equations-of-platformers/)）：

```
重力     g  = 2 * 跳躍高度 / 到頂時間²
起跳速度 v0 = -√(2 * g * 跳躍高度)     （y 向下為正）
```

### 2. 建議起始值（邏輯解析度 960×540、角色約 32×40px、60Hz 固定步進）

| 參數 | 建議值 | 說明 |
|------|--------|------|
| 跳躍高度 | 120 px（角色高的 3 倍） | 足以跳過 2 格障礙 |
| 到頂時間 | 0.38 s | → g ≈ 1660 px/s²、v0 ≈ -630 px/s |
| 下落重力倍率 | 上升 g × 1.0，下落 g × 1.6 | 瑪利歐式「上慢下快」，落地乾脆不飄 |
| 最大落速 | 900 px/s | 必須封頂，否則高速穿牆且難反應 |
| 跑速上限 | 300 px/s | |
| 地面加速度 | 2400 px/s²（0.125s 到全速） | 起步要快，才「聽話」 |
| 地面減速度 | 3000 px/s²（放開 0.1s 停） | 停得比起步更快 |
| 空中加速度 | 地面的 0.65 倍 | 保留空中修正能力（Celeste 甚至給全額） |
| 轉向加速度 | 地面的 2 倍 | 反方向輸入時瞬間有力，手感關鍵 |

### 3. 寬容機制（全部必做）— 出自 [Celeste & Forgiveness](https://maddythorson.medium.com/celeste-forgiveness-31e4a40399f1) 與 [Maddy Thorson 的手感串文](https://threadreaderapp.com/thread/1238338574220546049.html)

| 機制 | 值 | 實作 |
|------|-----|------|
| **土狼時間（Coyote Time）** | 0.08~0.1s（Celeste 用 5 幀） | 離開平台後計時器內仍可跳：`if(grounded) coyote = 0.09; else coyote -= dt;` 跳躍條件改成 `coyote > 0` |
| **跳躍緩衝（Jump Buffer）** | 0.1~0.15s | 落地前按跳也算數（模板中的 `actionBuffer` 即此） |
| **可變跳高（Variable Jump）** | 提早放開 → `vy *= 0.45` | 只在 `vy < 0`（上升中）時剪速；輕點=小跳、長按=大跳 |
| **頂點懸浮（Apex Hang）** | \|vy\| < 40 時重力 × 0.5 | 頂點多停留幾幀，方便瞄準落點 |
| **角落修正（Corner Correction）** | 4~8 px | 跳躍撞到天花板邊角時，若水平推移 ≤8px 能通過，就自動推開繼續上升 |
| **碰撞箱偏小** | 受擊箱比圖形小 20~30% | 「差一點」判玩家贏；攻擊/吃道具箱則放大 |

碰撞用 AABB + 分軸移動（先 X 後 Y）即可，注意 `grounded` 只在 Y 軸下移被擋時設 true。更多細節見 [2dengine 平台物理](https://2dengine.com/?p=platformers)。

### 4. 關卡設計：起承轉合（Kishōtenketsu）

任天堂的四步關卡結構（[Game Developer: The secret to Mario level design](https://www.gamedeveloper.com/design/the-secret-to-i-mario-i-level-design)、[GMTK: Super Mario 3D World's 4 Step Level Design](https://www.youtube.com/watch?v=dBmIkEvEBtA)）：

1. **起（介紹）**：在零風險環境展示新機制（第一個坑很淺、第一個敵人前有空地觀察）。
2. **承（發展）**：同一機制加上真實danger（坑變深、敵人變兩隻）。
3. **轉（翻轉）**：意外的組合或反用（機制+移動平台、機制倒過來用）。
4. **合（總結）**：一段展示完全掌握的高潮，然後乾脆地結束。

配套原則：

- **用場景教學，不用文字**：Mario 1-1 的第一隻栗寶寶迎面走來，玩家自然學會跳。把「必學動作」設計成前進的唯一路徑。
- **安全練習區**：新機制第一次出現時，失敗不致死。
- **檢查點**：每 30~60 秒遊玩放一個檢查點；重生瞬間完成（<1 秒），且重生點前不留無聊路段。
- **難度鋸齒**：難段 → 喘息段（獎勵、風景）→ 更難段。無盡型遊戲則用時間函數緩慢提升生成速度/密度，並設上限。

| **深海探險** | 潛水、解謎 | 深海 `#0b1e3f`・海藍 `#14476b`・水青 `#2e9cca`・泡沫 `#a8e6e2`・珊瑚橘 `#ff8c61`・魚黃 `#ffd447` |

用法：最深色當背景、最亮色當 UI 文字、**飽和度最高的一色只留給玩家角色**（畫面上最顯眼的必須是主角）。

### 無圖片資源的角色與場景技巧

- **有個性的 canvas 角色**：圓角矩形身體 + 兩顆大眼睛（白圓+黑瞳）即可有靈魂。眼睛看向移動方向、眨眼（每 3 秒把眼睛壓成一條線 0.1 秒）、待機呼吸（`sy = 1 + Math.sin(t*4)*0.03`）。
- **可讀剪影**：把角色填成純黑看剪影，仍要能認出頭/身/腳。敵人剪影必須與主角明顯不同（尖刺 vs 圓潤）。
- **視差背景（Parallax）**：3 層即可——遠山 `x - cam.x*0.2`、中景 `*0.5`、前景 `*1.0`。每層用同色系不同深淺，遠層淡、近層深。
- **光影便宜貨**：
  - 漸層天空：`ctx.createLinearGradient` 上淺下深。
  - 發光體：`ctx.shadowBlur = 20; ctx.shadowColor = 同色`（畫完關掉，效能貴）。
  - 全場氛圍：世界畫完後蓋一層 `rgba(20,10,60,0.15)` 統一色調。
  - 地面陰影：角色腳下畫半透明橢圓，跳越高橢圓越小——白送的高度感。
- **飄字（Floating Text）**：得分 `+100` 從事件點往上飄 40px 並淡出 0.6s。

## (f) 行動裝置支援

必做清單：

1. **viewport 與觸控鎖定**：模板已含 `user-scalable=no`、`touch-action:none`、`touchstart` 加 `{passive:false}` + `preventDefault()`（防雙擊縮放與捲動）。
2. **devicePixelRatio 縮放**：模板 `resize()` 已處理；記得 dpr 上限 2。
3. **觸控方案選擇**（依遊戲選一種，越簡單越好）：
   - **單鍵遊戲**（Flappy 類、跑酷）：全螢幕任一處點擊 = 動作鍵。首選。
   - **左右+跳**（平台跳躍）：左下角「◀ ▶」兩個大按鈕區（各 ≥ 螢幕寬 15%、畫半透明圓當提示），右半螢幕整片 = 跳。**用觸控區域而非畫面上的小按鈕**——拇指會飄，判定區要比圖示大 1.5 倍。
   - **自由移動**（俯視角）：虛擬搖桿——`touchstart` 位置為搖桿中心（浮動式，不固定在角落），拖曳向量即方向，超過 60px 取全速。比固定搖桿容錯高。
   - **滑動（swipe）**：只適合離散方向指令（2048、貪食蛇轉向），判定：位移 >30px 且取 |dx|/|dy| 較大軸。動作遊戲不要用 swipe，延遲感重。
4. **多點觸控**：務必遍歷 `e.touches` 全部觸點（如模板），否則「按著方向再按跳」會失效。
5. **直橫向**：遊戲鎖定一種比例設計（16:9 橫向為主），另一向用 letterbox 置中即可；若偵測到直立手機，可顯示「請轉橫」提示。
6. **UI 尺寸**：可點擊目標 ≥ 48px（實體約 9mm）；重要文字 ≥ 20px 邏輯像素。
7. **效能**：粒子上限（如 300 顆）、避免每幀 `new`、`shadowBlur` 節制使用；在中階手機測到穩定 60fps 才算過關。

## (g) 出貨前檢查清單

**手感**
- [ ] 跳躍有 coyote time、jump buffer、可變跳高（若為平台遊戲）
- [ ] 每個玩家動作都有音效 + 視覺回饋（對照 (c)7 配方表）
- [ ] 命中有 hit-stop 或震動；死亡有戲劇性（粒子+音+停頓）
- [ ] 鏡頭與 UI 數值都有緩動，無瞬移

**設計**
- [ ] 開場 30 秒內不看文字就會玩（找一個沒看過的人測）
- [ ] 難度曲線：第一分鐘不會死，之後穩定變難，有喘息段
- [ ] 死亡到重玩 < 1 秒；遊戲結束畫面顯示分數/最佳分數（`localStorage` 存最佳）
- [ ] 瑪利歐標準四項全過：即時回饋・漸進難度・失敗不挫折・一眼看懂

**技術**
- [ ] 單一 HTML 檔、零外部請求（斷網開啟仍可玩）
- [ ] 固定時間步進；切分頁再回來不會爆衝（acc 上限）
- [ ] 手機 Safari + Chrome 實測：觸控可玩、無捲動/縮放誤觸、60fps
- [ ] 各種螢幕比例（手機直橫、桌機）畫面不變形、不裁掉重要內容
- [ ] 音效在使用者首次互動後才啟動（AudioContext 政策），且音量 ≤ 0.2 不刺耳
- [ ] 截圖縮小 1/4 仍能辨識主角/敵人/道具
- [ ] 玩 3 分鐘無 console 錯誤、無記憶體持續成長（粒子/實體有清除）

**內容（營隊情境）**
- [ ] 標題、按鈕、提示皆為繁體中文，適合兒童
- [ ] 失敗文案正向鼓勵（「再挑戰一次！」而非「你輸了」）
