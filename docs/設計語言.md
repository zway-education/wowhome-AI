# AI未來創作營隊 設計語言（Design Language）

> 本文件記錄本 repo 的設計語言與品質底線。**之後在此 repo 製作的所有作品（遊戲、動畫、頁面）都必須符合本文件標準。**
> 搭配 `docs/遊戲設計指南.md`（做法與程式骨架）一起讀：指南教你怎麼做，本文件定義做到什麼程度才算合格。

---

## 1. 品牌視覺識別

### 品牌名稱
**AI未來創作營隊** — 頁面標題、頁尾必須出現。

### 品牌配色
| 用途 | 色碼 |
|------|------|
| 主色（青） | `#38e1ff` |
| 主色（藍） | `#4f7cff` |
| 主色（紫） | `#9a6bff` |
| 強調（粉） | `#ff6cc7` |
| PRO 金色系 | `#ffcf5c` → `#ffa72e` → `#ff7e3d` |
| 深色背景 | `#070b14`（PRO 頁 `#0a0805`） |

- 品牌標題用主色三色漸層文字（92°）
- 品牌圖示：圓角方塊漸層底＋白色機器人頭（SVG繪製，見 index.html 的 `.logo`；favicon 同款 data URI SVG）

### 頁面風格
- 深色為主、支援淺色模式（`prefers-color-scheme`）
- 卡片：半透明深色底＋細邊框＋圓角16px、懸停上浮＋主色描邊
- 背景：**預先計算的靜態多層 radial-gradient**（禁止即時 blur/backdrop-filter 大面積使用——會讓滑動卡頓）
- UI 介面文字不用表情符號；圖示用 SVG 繪製
- 卡片封面一律是**作品進行中的實際畫面截圖**，絕不出現文字色塊佔位

## 2. 作品品質底線（瑪利歐標準）

每個作品出貨前四原則缺一不可：
1. **即時回饋** — 任何輸入 <100ms 內有視覺＋聽覺回應
2. **漸進難度** — 前30秒不用讀字就能學會；難度隨時間/關卡緩升
3. **失敗不挫折** — 死亡後 <1 秒重來；失敗原因一目了然
4. **一眼看懂** — 主角/敵人/道具/背景視覺分層清楚，縮圖也能辨識

### 遊戲時長體驗（第五原則）
- 一場遊戲的節奏要適中：**不能開場10秒就死**（開頭要有安全學習期），也**不能拖到玩不完**
- 街機型（計分生存類）：一場典型時長落在 **1–3 分鐘**，難度曲線保證多數場次在 4 分鐘內自然收尾
- 關卡型：單關 **30–90 秒**，通關有結算畫面
- 經營/放置型：要有明確的階段目標與結局（例如 7 天挑戰＋評級），不能無止盡
- 動畫：總長 15–40 秒

## 3. 操作標準（不合格＝封鎖級缺陷）

- **操作說明必須顯示在標題畫面**：一至兩行＋圖示，分別說明電腦（按鍵）與平板（觸控）玩法；開玩後前幾秒再給提示浮層
- 輸入反應 <100ms，每個操作有即時回饋
- 預設操作直覺化：能單指點/拖就不用複雜手勢
- 寬容設計：判定範圍放大、跳躍有 coyote time 與輸入緩衝
- 虛擬搖桿/按鈕：有死區、跟隨拇指落點、置於平板雙手握持的左右下角、觸控目標 ≥48px、不遮擋遊戲重點區
- `touch-action: none` ＋ preventDefault：遊玩中頁面絕不誤觸捲動/縮放
- 鍵盤（電腦）與觸控（平板）**同時完整支援**；直向與橫向都能玩
- **可玩性實測**：出貨前用無頭瀏覽器模擬按鍵＋觸控，驗證遊戲狀態真的改變（角色會動、分數會變）

## 4. 美術標準

- **2D**：角色多部件繪製（輪廓、漸層陰影、表情、動作影格）——禁止純圓形/方形＋單色的陽春畫面
- **3D/方塊風**：要有透視或等角投影、每面受光陰影、環境光遮蔽感——禁止純色塊方塊人
- 背景要有層次：視差捲動、漸層天空、環境細節（雲、星、樹、觀眾等）
- 配色成套（參考遊戲設計指南的調色盤），剪影可辨識
- 可套用 **OpenMoji** 開源向量圖庫（4495張專業SVG，CC BY-SA 4.0）：
  - 取得：`npm i openmoji` → `node_modules/openmoji/color/svg/<HEXCODE>.svg`，用 `data/openmoji.json` 的英文 annotation 查表
  - 使用時把 SVG 內容直接內嵌進 HTML，並在檔案開頭加註 `<!-- 部分圖示：OpenMoji (CC BY-SA 4.0) openmoji.org -->`

## 5. 手感與音效標準

- 手感（juice）標配：關鍵事件粒子特效、受擊畫面震動、大招 hit-stop、角色 squash & stretch、UI 緩動轉場
- 音效一律 **Web Audio API 合成**（禁止外連音檔）：跳躍/擊中/得分/勝敗至少四種
- 必須有**靜音切換鈕**；動畫類預設靜音（瀏覽器擋自動播放），開頭提示「點聲音鈕開啟音效」

## 6. 動畫（影片類）標準

- 三幕結構：開場（場景與角色）→ 發展 → 高潮結尾；總長 15–40 秒
- 動作用緩動曲線（禁純線性）、有預備動作與跟隨
- 運鏡：關鍵節拍推鏡/搖鏡/震動；場景轉換有轉場
- 字幕有樣式與淡入淡出、小螢幕可讀、離邊緣有安全距離
- 結尾顯示**重新播放**按鈕（≥48px）

---

## 7. 技術規範

- 一個作品＝**一個完全自足的 HTML 檔**：CSS/JS 全部內嵌、零外部資源（無 CDN/字型/外連圖/音檔）
- 固定時間步進遊戲迴圈＋狀態機（標題→遊玩→結束）；`requestAnimationFrame`
- 畫布隨視窗縮放、`devicePixelRatio` 對應（高解析不糊）
- 最高分用 `localStorage` 記錄
- UI 文字用繁體中文
- 作者標記：`<head>` 加 `<meta name="author" content="名字">`
- PRO 版本迭代要在 `<!DOCTYPE html>` 後標記版本：`<!-- 專業AI升級 v2 -->`

## 8. 出貨前 QA 清單

```bash
# 1. JS 錯誤掃描（不得有任何 uncaught/syntax/reference error）
CHROME --headless=new --disable-gpu --no-sandbox --virtual-time-budget=5000 \
  --enable-logging=stderr --dump-dom "file://<檔案>" 2>&1 >/dev/null | grep -iE "uncaught|syntaxerror|referenceerror"
# 2. 零外部資源
grep -nE "https?://" <檔案>   # 只允許出現在授權註解文字中
# 3. 可玩性實測：模擬點擊+按鍵+觸控，斷言遊戲狀態改變
# 4. 檔案完整：結尾是 </html>，無截斷
```

## 9. 統計資料持續性（絕不可打破）

瀏覽/遊玩/投票統計存在外部計數服務（Abacus），識別碼由**作品檔名**經 `scripts/slug.js` 產生。
版本更新只改檔案內容、不改檔名，數據自動跨版本累積。因此：

- **禁止**更改 `scripts/slug.js` 的演算法（會讓所有統計歸零）
- **禁止**更改頁面 JS 中計數 key 的推導方式（`item.thumb` 去副檔名）
- **禁止**隨意重新命名作品檔名；真的必須改名時，要向使用者說明統計會重新開始
- 投票採「反向計數」：取消/改投是對 `-c` 取消計數器 +1，顯示票數＝投票數−取消數，永遠只用加法

## 10. 資料夾與流程規範

- 原作放 `games/`、`videos/`；專業升級版放 `pro/games/`、`pro/videos/`（同檔名）
- `index.html`、`home.html`、`pro.html`、`manifest.json`、`previews.json`、`thumbs/` 由系統自動維護，**不要手動修改**
- 作品 push 到 `main` 後，GitHub Actions 自動：截遊戲中畫面當封面 → 產生內嵌預覽 → 更新清單
- PRO 版採「討論→實作→QA」迭代流程：每輪必須經 遊戲設計師／10歲玩家／平板UX／操作手感 四視角討論並留下結論
